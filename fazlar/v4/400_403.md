Canlı 8080 üzerinde curl ile tekrarlanabilir kanıt toplandı. Özet ve rapor aşağıda.

Kanıtlı bulgu (bu ortamda curl ile)
1) Worker ile aynı şekil — tek JSON nesnesi

Dosya: .cursor-lora-test.json içeriği:

{"id":0,"path":"C:\\nonexistent\\missing.gguf","scale":1.0}

HTTP 400
{"error":{"code":400,"message":"Request body must be an array","type":"invalid_request_error"}}
2) Aynı uç, gövde JSON dizisi

[{"id":0,"path":"C:\\nonexistent\\x.gguf","scale":1.0}] → 400 değil, bu örnekte 500 invalid adapter id (dosya yolu sahte olduğu için ayrı konu).

Tek cümlelik teknik açıklama
/lora-adapters 400’ü, bu çalışan Qwen/llama sürecinde, gönderilen gövdenin tek JSON obje olması ve sunucunun “Request body must be an array” ile reddetmesi nedeniyle oluşuyor; bu hata LoRA dosyasının GGUF içeriğini okumadan üretiliyor.

Yani bu 400, “yanlış LoRA GGUF” kanıtı değil — önce istek şeması düzeltilmeden dosya türü tartışması yapılamaz.

Kodla uyum
packages/qa-sandbox/worker/r3mes_qa_worker/llama_client.py şunu gönderiyor:

body = {"id": slot_id, "path": adapter_path, "scale": scale}
Bu, yukarıdaki kanıtlanmış 400 ile çakışıyor.

apps/ai-engine içindeki _post_lora_adapter da aynı obje şemasını kullanıyor; aynı binary’de ai-engine üzerinden lora hot-swap da aynı hatayı alabilir (ayrı test).

Canlı bir 403 yanıtına veya worker loguna buradan erişemiyorum; message alanını siz çıkarmadan kök nedeni kesin seçemem. Aşağısı teşhis haritası + sizin dolduracağınız rapor şablonu + (gerekirse) tek satırlık minimal düzeltme önerisi.

2. Path / trailing slash (sık neden)
Hook yalnızca url === "/v1/internal/qa-result" iken ham gövdeyi okur. İstek /v1/internal/qa-result/ ise koşul tutmaz → rawBody set edilmez → genelde Ham gövde eksik.

Kontrol: Worker’daki tam URL’de sonda / yok mu?

3. İzlenecek sıra (sizin ortamda)
403 yanıtının tam JSON gövdesini kopyala → message’i yukarıdaki tabloyla eşleştir.
Aynı isteğin zaman damgasıyla backend access log satırını bul.
Worker + API: R3MES_QA_WEBHOOK_SECRET aynı mı (boşluk/satır sonu).
URL: POST …/v1/internal/qa-result — trailing slash yok.
Worker: HMAC = HMAC-SHA256(secret, **tam gövde baytları**) hex; backend ile aynı JSON serileştirmesi (tek boşluk farkı bile Geçersiz HMAC).
Düzelt → 200 veya aynı gövdeyle duplicate 200 → sonra DB: QaWebhookReceipt, Adapter.status, benchmarkScore.

5. Tek cümle şablon (message bilindikten sonra)
Örnek: “QA webhook 403, backend’in Geçersiz HMAC döndürmesi: worker ile API arasında aynı ham JSON baytı üzerinden HMAC hesaplanmadığı için (veya secret farklı).”

Örnek: “QA webhook 403, Ham gövde eksik: istek /v1/internal/qa-result/ path’i ile geldiği için ham gövde hook’u çalışmadı.”

6. Minimal düzeltme önerisi (yalnızca teşhis Ham gövde eksik + trailing slash ise)
Davranış: Worker URL’inden sondaki / kaldırılsın veya API tarafında path normalize edilsin (hook’ta url için trailing slash kırpma). Bu, kod değişikliği gerektirir; önce worker URL’ini düzeltmek genelde yeterlidir.

Secret / HMAC için kod değişikliği gerekmez — env ve payload hizası yeterli.

Başarı: 403 gövdesindeki message’i paylaştığınızda, hangi sınıfta olduğunuz ve sıradaki tek aksiyon (secret / header / path / JSON baytı) tek cümlede netleşir; canlı gövde olmadan burada o cümleyi sizin adınıza kilitleyemem.
